# プレイブック: Gmail → 選考状況の自動同期

**実行者: Claude Code（サブスク。API課金なし）。** 人間の操作は「結果（履歴）の確認」だけ。

Claude Code にこのファイルを渡して「gmail-sync を実行して」と言うと、以下を行う。
トークン節約のため **(1) 差分だけ読む (2) ルールで決まるものは Claude を使わず即更新** を徹底する。

## 手順

1. **対象スレッドを差分取得**
   - 直近の同期時刻を確認: `node tools/jobctl.mjs log 1` の一番上の日時、なければ過去14日。
   - Gmail を検索（MCP `search_threads`）。クエリ例:
     `newer_than:14d (選考 OR 面接 OR 説明会 OR ES OR エントリー OR 内定 OR お見送り OR 結果)`
   - 既に登録済み企業名（`node tools/jobctl.mjs list`）を優先的に突き合わせる。

2. **企業を特定**
   - 差出人ドメイン・署名・本文から企業名を推定し、`jobctl find "<名前>"` で既存行を探す。
   - 該当なしで明らかに新規選考なら `jobctl add-company "<名前>" --source gmail` で追加。

3. **ステータス判定（ルール優先＝トークン節約）**
   下表で決まるものは本文を熟読せず即適用する。複数該当時は「より先の段階」を採用。

   | 本文キーワード | → ステータス |
   |---|---|
   | お見送り / 残念ながら / 今回は見送 | 見送り |
   | 内定 / 採用 / オファー | 内定 |
   | 最終面接 | 最終面接 |
   | 二次面接 / 2次 | 二次面接 |
   | 一次面接 / 1次 / 面接のご案内 | 一次面接 |
   | Webテスト / 適性検査 / SPI / 玉手箱 | Webテスト |
   | ES / エントリーシート 提出 受付 | ES提出 |
   | エントリー 完了 受付 | エントリー済 |

   判定できない曖昧メールだけ、Claude が本文を読んで決める。

4. **反映（証跡つき）**
   - ステータス: `node tools/jobctl.mjs status "<id|名前>" "<状態>" --source gmail --evidence "<件名>"`
   - 面接/説明会など日時付き: `node tools/jobctl.mjs add-task "<id|名前>" "<一次面接 等>" <YYYY-MM-DD> --source gmail --evidence "<件名>"`
   - **後退する更新はしない**（例: 既に「二次面接」を「ES提出」に戻さない）。jobctl は同じ値なら自動でスキップする。

5. **報告**
   - 変更点を箇条書きで要約し、最後に `node tools/jobctl.mjs log 20` を実行して履歴を見せる。
   - 人間はポータルの「履歴」タブ（または上記ログ）で内容を確認する。これが「AIの出力チェック」。

## 機械化（さらにトークン削減）
Gmail 側でフィルタを作り、選考メールに自動でラベル（例 `就活`）を付けておくと、
手順1のクエリを `label:就活 newer_than:14d` に絞れて読む量が減る。

## 定期実行
`/schedule` で毎朝このプレイブックを回すルーティンを作れば、人間の操作なしで同期され、
ポータルを開くだけで最新になる。
